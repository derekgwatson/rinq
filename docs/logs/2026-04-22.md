## Decisions and WHYs

### "Call cutting out" was a visibility bug, not a call-quality bug
Sandrene messaged about calls to 0437707720 (Jessica Rachel) cutting out around 1:15pm AEST. Logs showed 7 attempts between 13:15‚Äì13:41 AEST (Sandrene x4, Stephanie x3), every one transitioning `initiated ‚Üí ringing ‚Üí busy` within ~1 second ‚Äî carrier returning busy on every try. Other outbound calls in the same window connected fine, so Rinq's trunk was healthy. The customer's end was the problem.

What Sandrene experienced ("cutting out") was the UX of a 1-second busy: ringtone starts, then gone. The feedback existed server-side (`call_log.status='busy'`), and the pipeline was ALREADY fully wired ‚Äî `/api/voice/call-outcome` endpoint at routes.py:4066, and `endCall()` already fetched and displayed it. The gap was purely visibility: a 4-second flash in the small `phoneStatus` text. Staff blew past it between redials. Same lesson as the 2026-04-20 mute/hold banner work: state that makes a call feel broken needs unmissable UI, not a status-line flash.

### Outbound outcome banner placed outside `.call-info`
`.call-info` is hidden via `display:none` on hangup (class `active` is removed). Anything inside it ‚Äî including the existing `#callNotice` used for transfer failures ‚Äî vanishes exactly when we want the outcome shown. Put the new `#outboundOutcome` element above the dialpad, outside `.call-info`, so it persists across the hangup. Auto-dismisses at 20s, clears on next `makeCall()`, and has a manual √ó (`dc7724e`).

### Email via Mabel, not a new Google Workspace client in Rinq
Derek was over the Resend free quota. The obvious shift was to Watson's Google Workspace SMTP, but hardwiring Workspace credentials into Rinq couples two things that don't need coupling: Rinq should stay tenant-agnostic where practical, and Workspace credentials belong with the account that owns them (Watson). Mabel already exists in bot-team for exactly this ‚Äî a Flask microservice at `smtp.gmail.com:587` sending from `mabel.bot@watsonblinds.com.au`, auth via shared `BOT_API_KEY`. Same pattern as Clara (customers) and Otto (orders). No Watson-specific credentials live in Rinq.

New `WatsonMabelEmailService` at `rinq/integrations/watson/email.py` (`3587d0e`). Reuses `BotHttpClient` + `get_bot_url('mabel')`. Recording_service.py signatures already matched Mabel's `EmailRequest` (pydantic) shape exactly ‚Äî `to`, `subject`, `text_body`, `attachments` (filename/content_type/content_base64), `metadata` ‚Äî so no other code changes were needed. Provider selection: auto-selects Mabel when `WATSON_MABEL_URL` set, otherwise Resend if `RESEND_API_KEY` set. Override with `RINQ_EMAIL_PROVIDER=mabel|resend`.

### Global email provider, not per-tenant ‚Äî for now
The current `init_integrations()` picks a single `_email_service` at app startup. Making that per-tenant is straightforward code-wise (read a tenant column), but Derek flagged the decision as bigger than coding: per-tenant provider selection has IP-ownership implications in the multi-tenant SaaS model that haven't been resolved yet. Keeping it global is the right move until that conversation happens. Watson is ~95% of Rinq's email volume anyway; flipping the global to Mabel solves the quota pain immediately.

### GitHub repo moved from personal to work account
`dezgo/rinq` ‚Üí `derekgwatson/rinq`. The `github-work` SSH host was already configured locally pointing at the right identity. Empty commit `c4bd888` verified the new remote + deploy workflow end-to-end.

## Key discoveries

- **Outcome pipeline was already wired ‚Äî just invisible.** Future "staff missed this" reports should first ask: does the feedback mechanism exist at all, or does it just fail to draw the eye? The diagnostic order matters.
- **GitHub user-to-user repo transfers require recipient acceptance via email.** Org-to-user and owner-initiated transfers don't. The GitHub UI doesn't make this obvious ‚Äî initial attempt looked like the transfer had failed.
- **Deploy keys and repo secrets transferred automatically** with the user-to-user repo move. The GitHub docs imply secrets don't transfer across accounts, but `SSH_PRIVATE_KEY` was intact on `derekgwatson/rinq` post-transfer.
- **Mabel is a drop-in for the Clara/Otto pattern.** `BotHttpClient` + `WATSON_{NAME}_URL` scaffolding makes a new bot integration ~50 lines when the shape fits.
- **Mabel already validates via pydantic `EmailRequest`.** Her schema matched what `recording_service.py` was already passing ‚Äî no adapter logic needed on either side.
