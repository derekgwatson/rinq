## What was done

### 1. Tour fix: Transfer step highlight (`rinq/web/templates/phone.html`)

The guided tour's Transfer step pointed to `#btnTransfer`, which is hidden outside of active calls. Driver.js showed an orphaned popover with nothing highlighted.

**Fix:** Added a `showCallBarDemo: true` flag to the Transfer step. In `onHighlightStarted`, the handler checks this flag and temporarily adds `.active` to `#callInfo`, making `#btnTransfer` visible and highlightable. The class is removed on the next step and in `onDestroyed`.

### 2. Tour fix: Demo panel index lookup (`rinq/web/templates/phone.html`)

`onHighlightStarted` was using `filtered.indexOf(step)` to find the current step and activate the sidebar demo panel. This always returned -1 because Driver.js callbacks receive a mapped step config object (different reference), not the original object from the `filtered` array.

**Fix:** Replaced `filtered.indexOf(step)` with `driverObj.getActiveIndex()`.

### 3. Bug fix: "A Call is already active" after warm transfer hand-off (`rinq/web/templates/phone.html`)

**Root cause:** `completeTransfer()` called `endCall()` (which nulls JS state variables) without ever calling `currentCall.disconnect()` on the Twilio Device SDK. The WebRTC connection stayed alive. The next `device.connect()` call (answering the next queue call) threw "A Call is already active". A hard refresh fixed it because it recreated the Device fresh.

**Fix:** Capture `callRef = currentCall` before `endCall()`, then call `callRef.disconnect()` after.

### 4. UX fix: Replace `alert()` with `showPhoneError()` (`rinq/web/templates/phone.html`)

Added `showPhoneError(msg)` helper that flashes an error in the status bar for 5 seconds then reverts to Ready. Replaced all `alert()` calls in the softphone `answerQueueCall` function.
