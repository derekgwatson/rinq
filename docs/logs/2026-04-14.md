# 2026-04-14 Session Log

## Contacts directory dedup + hide_mobile

Cleaned up the phone home-page contacts list. In the Watson tenant each staff member was appearing twice — once from the staff source (Peter `/api/staff` or local `staff_extensions`) and once from the `address_book` table populated by the daily 2am Peter sync. Plus there was no way to hide your personal mobile from the directory, and the green call button was duplicating the blue extension button when a staff member had no mobile.

Four small commits, all on `main` and pushed.

### 1. `18de640` — Add `hide_mobile` setting

- Migration 068 — `staff_extensions.hide_mobile INTEGER DEFAULT 0`.
- New toggle on the home-page Settings card: "Hide my mobile number".
- `/api/contacts` suppresses the mobile when the flag is set.
- Defaults off so existing behaviour is unchanged.

### 2. `835c32a` — Extend `hide_mobile` to address book entries + drop `show_in_pam` toggle

The first commit only stopped the staff source from showing the mobile. The address book copy of the same person still leaked it.

- Migration 069 — `address_book.email TEXT` (NULL on existing rows, backfilled on next sync).
- `PeterAddressBookSource.get_entries()` now populates `email` from Peter's `google_primary_email` / `work_email` / `email` (in that order, since Peter's format varies).
- `db.upsert_address_book_entry()` takes a new `email` param and persists it.
- Removed the misleading "Show in staff directory" toggle from the home page Settings card. It only ever controlled `staff_extensions.show_in_pam`, which feeds Peter's PAM directory (`/staff-phones`), NOT Rinq's `/api/contacts`. `update_staff_extension` no longer writes the field. Column kept (Peter still reads it).

### 3. `1ce55af` — Dedupe staff against their address book copy + drop phone fallback

- `/api/contacts` now dedupes by email after assembling both sources. Staff entries take precedence; the staff card is enriched with `section` / `position` from the address book entry when missing.
- Dropped `phone = ... or ext.get('extension', '')` in both branches of the staff loop (the `if peter_staff: ... else:` branches). Without that fallback, the green phone-call button only appears when there's a real mobile and no longer duplicates the blue extension button.

### 4. `4076d33` — Enrich staff with address book `display_mobile` when staff source has none

The dedup in commit 3 was too aggressive: when the staff source had no mobile but the address book entry did, the merged record dropped the address book mobile. Restored it — the staff card now picks up `display_mobile` from the address book copy unless `hide_mobile` is set.

## Diagnosis notes

- **Two-source list:** `/api/contacts` has an `if peter_staff: ... else:` branch. When Peter's `/api/staff` is unreachable from the Rinq server (cross-server latency / outage), `peter_staff` is empty and the local `staff_extensions` fallback runs. That's how Derek sees his own entry as just "Derek" (name derived from email) in the OTHER section instead of via his rich Peter record. Not changed today.
- **`show_in_pam` is a non-obvious gotcha.** The home-page toggle labelled "Show in staff directory" had nothing to do with the directory you actually see in Rinq — it bound to a flag consumed only by Peter's `/staff-phones` API. Removed the toggle and added a CLAUDE.md gotcha so future-me doesn't re-bind to it.
- **Email key on address book was the unlock.** Without it the dedup couldn't run and `hide_mobile` couldn't reach the address book copy.
